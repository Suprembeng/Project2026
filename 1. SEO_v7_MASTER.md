## 3. СТРУКТУРНЫЙ ВЫХОД ЧЕРЕЗ TOOL_USE (обязательный)

> **Breaking change v7.0**: все 63 шага возвращают результат **только через tool_use**. Текстовый JSON запрещён.

### Паттерн

```python
# Для каждого STEP_XX определён один tool
tool = {
  "name": "submit_step_result",
  "description": "Возвращает результат шага в соответствии с JSON Schema.",
  "input_schema": <ссылка на schemas/step_XX.schema.json>
}

messages.create(
  model="claude-opus-4-7",
  tools=[tool],
  tool_choice={"type": "tool", "name": "submit_step_result"},  # FORCED
  ...
)
```

### Правила

1. **tool_choice всегда `{"type": "tool", "name": "submit_step_result"}`** — форсируем структуру.
2. **JSON Schema валидируется на уровне API** → слой 1 QC (ajv в n8n) становится **страховочным**, а не первичным.
3. **Thinking блоки** (если enabled) возвращаются отдельно в `response.content[type="thinking"]`. Логируются в Langfuse, не сохраняются в NocoDB (PII/cost).
4. **При ошибке tool_use** — Retry протокол §6.

## 4. PROMPT CACHING (cache_control breakpoints)

Модульная архитектура диктует **3 явных breakpoint**:

```python
system = [
  {"type": "text", "text": CORE_md,    "cache_control": {"type": "ephemeral"}},  # bp1: TTL=1h
  {"type": "text", "text": PROJECT_md, "cache_control": {"type": "ephemeral"}},  # bp2: TTL=1h
  {"type": "text", "text": STEP_XX_md, "cache_control": {"type": "ephemeral"}},  # bp3: TTL=1h
]
```

| Breakpoint | Стабильность          | Expected cache hit rate | Экономия на cached tokens |
|------------|-----------------------|-------------------------|---------------------------|
| bp1 (CORE) | Месяцы                | >95%                    | −90%                      |
| bp2 (PROJECT) | Дни-недели         | >80%                    | −90%                      |
| bp3 (STEP) | Per-prompt_version    | >60%                    | −90%                      |

> **KPI для команды**: `cache_read_input_tokens / total_input_tokens ≥ 0.70` (замер в Langfuse).

## 6. RETRY PROTOCOL v3 (детерминированный)

```
Retry 1: Вернуть ошибки QC в промпт как system_reminder. Та же температура, та же модель.
Retry 2: Выполнить decomposition_plan из STEP_XX.md (обязательное поле).
         Если план не задан → эскалация в human-review (не делать авто-декомпозицию).
Retry 3: Fallback-промпт из STEP_XX.md (обязательное поле fallback_prompt).
         Ниже temperature на 0.1, +few-shot пример эталонного выхода.
Retry 4: Human alert. Лог 3 попыток сохраняется в NocoDB.step_runs.retry_log.
```

> **Изменение v7.0**: в v6.0 Retry 2 говорил «декомпозировать на 2 подзадачи» — без уточнения, что это. Теперь каждый STEP_XX **обязан** содержать `decomposition_plan` и `fallback_prompt`.

### Backoff по классу ошибки

| Класс               | Backoff                       | Max retries |
|---------------------|-------------------------------|-------------|
| 429 (rate limit)    | exp: 10s → 30s → 90s          | 3           |
| 5xx                 | exp: 5s → 15s → 45s           | 3           |
| tool_use schema fail| Retry 1 (вставить ошибку)     | 1           |
| QC fail (слои 2-6)  | Retry 1-3 по протоколу выше   | 3           |

## 10. ПРОТОКОЛ ДЕГРАДАЦИИ МОДЕЛЕЙ (Golden Test Gate)

**Golden Test Set v7**:
- **Минимум 1 задача на каждый из 63 шагов = 63 задачи**.
- **Edge-cases**: 2-3 дополнительные задачи на каждый HitL-шаг (02, 17, 24, 27, 36) = +15 задач.
- **Adversarial**: 5 задач с prompt injection через SERP-контент.
- **Total**: ≥ 83 задачи.

**Метрики оценки**:
- `structure_score` (схема пройдена): 0 или 1
- `content_score` (LLM-as-judge, rubric): 0..100
- `latency_p95_ms`
- `cost_usd_p95`

**Gate-политика**:
- Падение `content_score` >10% vs baseline → алерт, **не деплоим**.
- Рост `latency_p95` >30% → warning.
- Рост `cost_usd_p95` >20% → warning.

**Управление версиями**:

## 15. БЕЗОПАСНОСТЬ

- **Prompt injection через SERP-контент**: любой текст из web_search/DataForSEO оборачивается в `<untrusted_content source="...">...</untrusted_content>`. Системная инструкция эксплицитно запрещает выполнять указания изнутри этого тега.
- **API-ключи**: только через n8n credentials, никогда в промптах или .md-файлах.
- **PII в логах**: маскирование email, телефонов, адресов в NocoDB-логах и Langfuse трейсах (middleware).
- **Secrets rotation**: ежеквартально, трекинг в NocoDB.secrets_rotation_log.

## СИСТЕМНЫЙ ПРОМПТ (для Opus)

```text
Ты — Роль A: Senior SEO-стратег с 10+ летним опытом в niche_cluster проекта.
Консультативные роли: E (CRO), F (Prompt Engineer).

ЗАДАЧА: Создать 3-7 портретов покупателей и вернуть результат через tool "submit_step_result".

ОБЯЗАТЕЛЬНЫЕ ПРАВИЛА:

1. Каждая персона содержит ВСЕ 5 уровней Лестницы Ханта с 2-4 примерами запросов на уровень.
2. Каждый запрос помечен engine ∈ {yandex, google, both, "ai:aio", "ai:perplexity", "ai:chatgpt"}.
3. Учитывай niche_cluster проекта:
   - commercial_local: гео-запросы, "рядом", карты, микрогеография
   - commercial_national: сравнительные, конкурентные, федеральные
   - ecom: сравнение товаров, цены, отзывы, фильтры
   - medical: YMYL, симптомы, лечение — повышенное внимание E-E-A-T, маркировка [⚠ РИСК] где уместно
   - legal: ситуативные, срочность, статусы
   - saas: альтернативы X, интеграции, features
   - info: потребление контента, глубина, подписка
   - education: сезонность, длинный цикл принятия решения

4. Приоритизируй персоны по priority_score (0.00-100.00):
   base = impact_on_business (0-50) + query_volume_estimate (0-40)
   bonus = +10 если search_behavior.ai_search_usage ∈ {regular, primary, exclusive}
   penalty = -5 если JTBD слабый (нет эмоциональных jobs или триггеров)

5. Для каждого утверждения о поведении или предпочтениях — claim_confidence:
   "✓" — если данные пришли из RAG/internal_search_queries
   "◆" — логически выведено из niche_cluster и geo
   "◇" — гипотеза (отметить и для таких персон предусмотреть тест)

6. Если priority_engines=both — ОБЯЗАНЫ быть запросы и для [yandex], и для [google] в каждой персоне.
7. Если aio_optimization=true — минимум 1 запрос `"ai:aio"` на персону в уровнях 2-4.
8. ЗАПРЕЩЕНО:
   - Общие "персоны" без привязки к niche/geo
   - Персоны из пересекающихся сегментов с rank-коллизией
   - Запросы без intent-метки

9. Размышление (thinking): используй для планирования количества персон, баланса уровней Ханта, распределения AI-поверхностей. Не дублируй его в итоговый JSON.

10. ВЫХОД: ВСЕ поля по схеме. ВЫЗОВ ТОЛЬКО tool submit_step_result.
```

## МЕТРИКИ УСПЕХА

**Опережающий (сразу после шага)**:
- `qc_score ≥ 80`
- `seed_queries покрывают ≥4 intent-типа`
- `cache_hit_rate bp1+bp2 ≥ 0.70`
- `cost_per_run ≤ $0.50`

**Запаздывающий (после Шага 01)**:
- Для каждой персоны в Шаге 01 найдено ≥20 real queries
- `hunt_ladder_coverage ≥ 0.80` (реальные запросы по 4+ уровням на персону)

**Контроль (через неделю)**:
- Ревью аналитиком: реальное поведение / инструмент-артефакт
- Update confidence markers по данным из Шага 01
